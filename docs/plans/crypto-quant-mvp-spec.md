# Plan: 构建自用型币圈量化交易 MVP 系统，使交易者能在 Hyperliquid 永续 + Binance 合约上完成 数据→回测→模拟盘→实盘→监控 全闭环

**Complexity:** Large — 全新子系统，6+ 核心模块跨数据/策略/执行/风控/安全/监控多个领域，接口必须先稳定再实现。

## Goal

Enable 有经验的个人交易者 在本地运行一套 Python 量化系统，覆盖 Hyperliquid 永续合约和 Binance U 本位合约，完成从历史数据获取、策略回测、模拟盘验证到小资金实盘的全闭环，且 API Key 和私钥永不离开本机。

## Deliverables

### 数据层
- **数据采集服务** `data/feed/`: 通过 Hyperliquid SDK 和 Binance REST/WebSocket API 获取 K 线（OHLCV）、资金费率、深度快照，统一为标准 DataFrame 格式  →  为策略引擎和回测引擎提供一致的输入数据
- **本地时序存储** `data/store/`: 基于 SQLite + parquet 的轻量历史数据存储方案，按交易对+时间框架分片  →  支持回测引擎快速加载历史数据，无需依赖外部数据库

### 策略引擎
- **策略基类** `strategy/base.py`: 抽象基类 `Strategy`，定义 `on_bar()`, `on_tick()`, `generate_signal()` 接口，内置指标计算钩子  →  所有策略继承此基类，保证策略层与执行层解耦
- **示例策略** `strategy/examples/`: 至少 2 个可运行的示例策略（双均线交叉 + 布林带突破），用于验证框架完整性  →  证明策略引擎可工作，为后续自定义策略提供模板

### 回测引擎
- **事件驱动回测核心** `backtest/engine.py`: 逐 bar 驱动，严格禁止前瞻偏差（bar 内只能访问当前及历史数据），内置手续费模型（Maker/Taker）和滑点模拟  →  回测结果可信，避免"回测赚钱实盘亏钱"
- **回测报告生成** `backtest/report.py`: 输出关键指标（总收益、最大回撤、夏普比率、胜率、盈亏比）+ HTML 图表  →  策略评估可视化

### 执行引擎
- **交易所适配器** `execution/adapters/`: 统一接口 `ExchangeAdapter`，Hyperliquid 适配器（基于官方 Python SDK，签名交易）和 Binance 适配器（基于 ccxt fork 审计版）  →  策略代码无需关心底层交易所差异
- **订单管理器** `execution/order_manager.py`: 订单生命周期管理（创建→提交→部分成交→全部成交/取消），幂等性保证，网络中断自动重试  →  实盘执行可靠，不因网络问题导致重复下单或丢失订单
- **模拟盘模式** `execution/dry_run.py`: 基于实时行情的纸上交易，不发送真实订单，但模拟撮合逻辑  →  实盘前的安全验证阶段

### 风控系统
- **风控引擎** `risk/engine.py`: 多层风控 — 单笔最大仓位限制、日最大亏损熔断、总资金利用率上限、滑点保护阈值  →  防止策略异常导致灾难性损失
- **资金隔离** `risk/allocation.py`: 每个策略实例独立资金池，互不影响  →  多策略运行时风险隔离

### 密钥安全
- **本地密钥管理** `security/key_manager.py`: API Key 和 Hyperliquid 私钥使用本地加密文件存储（Fernet 对称加密 + 主密码派生），密钥仅在内存中解密使用，不落日志  →  密钥永不离开本机，满足自用系统安全需求

### 监控与配置
- **配置系统** `config/`: YAML 配置文件，支持交易所参数、策略参数、风控阈值、日志级别  →  所有行为可配置化，无需改代码
- **日志与监控** `monitoring/`: 结构化日志（JSON 格式）+ 关键事件告警（Telegram Bot 通知）  →  7x24 运行可观测

### 项目基础设施
- **项目脚手架**: pyproject.toml + Docker Compose + Makefile + 目录结构  →  一键启动开发环境
- **集成入口** `main.py`: CLI 入口，支持 `backtest`, `dry-run`, `live` 三种运行模式  →  用户通过命令行切换模式

**Out of scope:**
- 现货交易（V1 阶段）
- 链上狙击 / Meme 币（V2 阶段）
- 多策略并发调度（V1 基础支持，完整调度 V2）
- 移动端 / Web UI（V2 阶段）
- 策略市场 / 信号市场（V2 阶段）
- ML/AI 策略集成（V1 阶段）

## Acceptance Criteria

```
AC-1  [Happy path - 回测]    Given 已下载 BTC/USDT 4h K线数据(≥1000 bars) When 用双均线策略运行回测 Then 输出包含总收益、最大回撤、夏普比率的报告，且回测过程中无任何前瞻偏差（验证方式：在 bar[i] 只能访问 bar[0..i] 的数据）
AC-2  [Happy path - 模拟盘]  Given 配置了 Binance 测试网 API Key When 以 dry-run 模式启动系统并接入实时行情 Then 策略产生信号后生成模拟订单记录，不发送真实订单，且订单状态正确流转
AC-3  [Happy path - 实盘]    Given 配置了 Hyperliquid 签名密钥 When 以 live 模式启动并对 Hyperliquid 下限价单 Then 订单成功提交并上链，订单状态正确同步
AC-4  [Error path - 风控熔断] Given 设置日最大亏损 5% When 模拟盘中策略连续亏损触发阈值 Then 系统自动停止该策略的所有新订单，发出告警，已有仓位保持不动
AC-5  [Error path - 网络中断] Given live 模式运行中 When Binance WebSocket 断连超过 30 秒 Then 系统自动重连，重连期间暂停新信号，已挂订单保持，恢复后正常继续
AC-6  [Edge case - 密钥安全] Given 系统启动并加载密钥 When 检查日志输出和进程内存 Then API Key 和私钥不以明文出现在日志中，加密文件使用主密码派生密钥保护
```

## Implementation Path

**Complexity: Large** — 全新多模块子系统，接口设计先行，增量实现。

### Phase 1: 基础设施与数据层（Week 1）
1. **项目脚手架搭建** — pyproject.toml、目录结构、Docker Compose、Makefile、CI lint → 文件: `pyproject.toml`, `docker-compose.yml`, `Makefile`, 全部 `__init__.py` → 验证: `make setup && make lint` 通过
2. **配置系统** — YAML 配置加载、环境变量覆盖、配置校验(pydantic) → 文件: `config/` → 验证: 加载示例配置无报错
3. **本地密钥管理** — Fernet 加密、主密码派生(PBKDF2)、密钥存取接口 → 文件: `security/key_manager.py` → 验证: 加密→解密→比对一致，日志中无明文密钥
4. **数据采集服务** — Hyperliquid SDK 获取 K 线、Binance ccxt 获取 K 线、统一 DataFrame 格式 → 文件: `data/feed/` → 验证: 获取 BTC/USDT 1h K 线 ≥100 bars 并存储
5. **本地时序存储** — SQLite 元数据 + parquet 数据文件、按交易对+时间框架分片 → 文件: `data/store/` → 验证: 写入→读取→数据一致

### Phase 2: 策略引擎与回测（Week 2-3）
6. **策略基类** — 抽象基类定义、指标计算集成(ta-lib/pandas-ta)、信号数据结构 → 文件: `strategy/base.py` → 验证: 子类可继承并实现 on_bar()
7. **示例策略** — 双均线交叉 + 布林带突破 → 文件: `strategy/examples/` → 验证: 两个策略均可生成信号
8. **回测引擎核心** — 事件驱动逐 bar 推进、前瞻偏差防护、手续费模型、滑点模拟 → 文件: `backtest/engine.py` → 验证: AC-1 通过
9. **回测报告** — 指标计算 + HTML 图表生成 → 文件: `backtest/report.py` → 验证: 报告包含所有关键指标

### Phase 3: 执行引擎（Week 3-4）
10. **交易所适配器接口** — 统一 ExchangeAdapter 抽象 → 文件: `execution/adapters/base.py` → 验证: 接口定义完整
11. **Binance 适配器** — 基于 ccxt，支持 U 本位合约下单/查询/撤单 → 文件: `execution/adapters/binance.py` → 验证: 测试网下单成功
12. **Hyperliquid 适配器** — 基于官方 SDK，签名交易 → 文件: `execution/adapters/hyperliquid.py` → 验证: AC-3 通过
13. **订单管理器** — 订单生命周期、幂等性、重试 → 文件: `execution/order_manager.py` → 验证: 网络中断后重试不重复下单
14. **模拟盘模式** — 实时行情 + 纸上撮合 → 文件: `execution/dry_run.py` → 验证: AC-2 通过

### Phase 4: 风控与监控（Week 4-5）
15. **风控引擎** — 仓位限制、日亏损熔断、滑点保护 → 文件: `risk/engine.py` → 验证: AC-4 通过
16. **资金隔离** — 策略级资金池 → 文件: `risk/allocation.py` → 验证: 两个策略资金互不影响
17. **日志系统** — 结构化 JSON 日志、密钥脱敏 → 文件: `monitoring/logger.py` → 验证: 日志中无密钥明文
18. **Telegram 告警** — 关键事件推送 → 文件: `monitoring/alert.py` → 验证: 风控触发时收到 Telegram 消息

### Phase 5: 集成与验收（Week 5-6）
19. **CLI 入口** — `main.py` 支持 backtest/dry-run/live 三种模式 → 文件: `main.py` → 验证: 三种模式均可启动
20. **端到端测试** — 完整流程: 数据下载→回测→模拟盘→(可选)实盘 → 验证: AC-1 至 AC-6 全部通过
21. **Docker 打包** — 生产镜像 + docker-compose 一键启动 → 验证: `docker-compose up` 即可运行

## Open Questions

  - 假设用户已有 Hyperliquid 钱包私钥和 Binance API Key
  - 假设本地运行环境为 macOS 或 Linux（Docker 可选）
  - 假设初期仅交易 BTC/USDT 和 ETH/USDT 主流交易对
  - 假设 Binance 使用 ccxt 库（fork 后审计 referral code）
  - 假设 Telegram Bot Token 由用户自行创建并提供
